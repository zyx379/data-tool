## 一、产品概述

### 1.1 背景

ZoeDevOps 当前具备两大核心能力：「数据查询」（浏览表结构、条件查询、导出数据）和「智能分析」（输入日志 ID → AI 自动排查问题）。然而运维人员在日常工作中还有大量**数据报表**需求——他们需要用自然语言描述需求，快速得到 SQL、查询结果和可视化图表，而不是手动编写 SQL、拼接 Excel。

### 1.2 目标用户

HIS（医院信息系统）运维人员、医院信息科工程师。他们熟悉业务但 SQL 能力参差不齐，需要低门槛的数据查询与报表工具。

### 1.3 解决的问题

| 痛点 | AI报表模块如何解决 |
|------|-------------------|
| 写 SQL 门槛高 | 自然语言输入，AI 自动生成 SQL |
| 不知表之间如何关联 | AI 自动发现、验证、缓存表关系 |
| 查询结果不够直观 | 自动推荐图表类型，可视化展示 |
| 临时数据需要手工写 INSERT | 拖拽 Excel/图片 → AI 生成插入语句 |
| 重复报表每次重写 SQL | 保存为模板，参数化一键复用 |
| 报表需求需反复沟通 | 对话式迭代，追问即修改 |

## 三、功能需求详述

---

### 3.1 智能生成（自然语言 → SQL + 图表）

#### 3.1.1 输入方式

用户在主界面「AI报表」标签页中，通过**对话输入框**描述报表需求，支持纯文本输入，也支持拖拽文件附件配合描述（见 3.3）。

**典型输入示例：**

> "查询本月门诊各科室的收入汇总，按收入降序排列"

> "统计最近一周药品消耗量前十的药品名称和数量"

> "查询2026年5月1日到5月15日住院患者的平均住院天数，按病区分组"

> "找出上月挂号量最多的前5位医生"

#### 3.1.2 理解意图

AI 收到用户描述后，应完成以下意图提取：

- **查询目标**：用户想看什么数据？（如：科室收入、药品消耗、住院天数）
- **筛选条件**：有哪些过滤条件？（如：本月、最近一周、2026年5月1日到5月15日）
- **聚合方式**：需要分组/求和/计数/平均值？（如：按科室分组、降序排列、TOP10）
- **时间范围**：涉及哪些时间字段？（如：本月 → 当月第一天到当天）
- **数据来源**：可能涉及哪些数据库表？（如：门诊收入表、药品消耗表、住院记录表）

AI 若无法确定某些信息，应在对话中主动向用户确认，例如：
> "您说的'本月'是指自然月（5月1日到今天）还是最近30天？"

#### 3.1.3 探索表结构

AI 应利用系统已有的 Schema 缓存（即「数据查询」模块加载的表结构 JSON 数据）来完成以下工作：

- 根据意图中的关键词（如"门诊""收入""药品""住院"），模糊匹配数据库中相关表
- 查看候选表的列信息（列名、数据类型、注释），确认哪些列可用于查询
- 将候选表及其列信息展示给用户，供用户确认或调整

#### 3.1.4 表关系验证（关键）

当查询涉及多张表（需要 JOIN）时，AI 必须遵循以下流程：

1. **先查缓存**：查询本地已存储的「已验证表关系」，看是否已有相关表的关联记录
2. **缓存命中**：直接使用已验证的关系生成 JOIN 条件
3. **缓存未命中**：AI 根据列名和注释推断可能的关联列（如：两个表都有 `patient_id`），然后**实际执行一条验证 SELECT** 来确认（如 `SELECT COUNT(*) FROM A INNER JOIN B ON A.col = B.col WHERE ROWNUM <= 10`）
4. **验证通过**：将关系记录到本地缓存（包含：主表、主表列、关联表、关联表列、验证 SQL、验证时间）
5. **验证失败**：尝试其他候选列，或向用户确认表关系

**缓存失效机制**：当用户重新加载 Schema（刷新表结构）时，该数据源下所有表关系自动标记为「待重新验证」。下次使用时会重新执行验证 SELECT，确认后恢复有效状态。

#### 3.1.5 SQL 生成

AI 基于以上所有信息生成 SQL 语句，必须满足以下约束：

- **只生成 SELECT 语句**，禁止 INSERT / UPDATE / DELETE / DROP / ALTER / TRUNCATE 等任何修改操作
- **必须限制返回行数**：Oracle 使用 `ROWNUM <= 500`，达梦使用 `TOP 500`，避免全表扫描返回海量数据
- **时间条件必须使用正确的数据库函数**：Oracle 用 `TO_DATE`，达梦用 `CAST`
- **表名和列名必须来自已验证的 Schema 缓存**，不能凭空编造
- 生成的 SQL 必须先**展示给用户预览**，用户确认后才执行

#### 3.1.6 查询执行

用户确认 SQL 后，系统执行查询：

- 展示执行耗时
- 展示返回行数（如超出限制，提示用户缩小范围或导出完整数据）
- 结果以表格形式展示，列头支持显示中文注释（复用数据查询模块的列名中文显示功能）

#### 3.1.7 图表生成

查询有结果后，AI 根据数据特征自动选择最合适的图表类型：

| 数据特征 | 推荐图表 | 示例场景 |
|----------|---------|---------|
| 时间序列趋势 | 折线图 | 每日门诊量变化 |
| 分类对比 | 柱状图 | 各科室收入对比 |
| 占比分布 | 饼图 | 各药品消耗占比 |
| 明细数据 | 表格 | 患者费用明细 |
| 相关性分析 | 散点图 | 住院天数与费用关系 |

**图表要求：**
- 使用 ECharts 渲染
- 支持图表类型切换（用户可在折线图、柱状图、饼图、表格之间切换）
- 图表标题使用报表标题
- 坐标轴标签清晰可读（长文本自动旋转或截断）
- 支持 Tooltip（鼠标悬停显示详情）
- 饼图显示百分比标签

#### 3.1.8 对话式追问

报表生成后，用户可以在对话输入框中继续追问，对当前报表进行微调：

**典型追问场景：**

| 用户追问 | AI 行为 |
|----------|--------|
| "改成按周统计" | 调整 SQL 的时间聚合粒度，重新查询 |
| "只看内科的数据" | 添加 WHERE 条件，重新查询 |
| "换成柱状图" | 切换图表类型，不重新查询 |
| "按收入降序排列" | 调整 ORDER BY，重新查询 |
| "导出为 Excel" | 触发导出功能 |
| "只看前5条" | 添加 ROWNUM/TOP 限制，重新查询 |

AI 应理解追问是对当前报表的修改，而非全新请求。追问时应保留之前的查询上下文。

### 3.2 表关系管理

#### 3.2.1 自动发现

AI 在报表生成过程中自动完成表关系的发现与验证（见 3.1.4），用户无需手动操作。

#### 3.2.2 表关系查看

在「AI报表」页面的侧边栏中，提供「表关系」查看入口：

- 显示当前项目下所有已验证的表关系列表
- 每条关系显示：主表 → 关联表、关联列、JOIN 类型、验证时间
- 有效关系显示绿色标记，待重新验证显示黄色标记
- 支持按表名搜索过滤

#### 3.2.3 表关系管理

用户可以对表关系进行以下操作：

- **手动添加**：已知两个表的关联关系，可手动录入（主表、主表列、关联表、关联表列、JOIN 类型）
- **手动验证**：对「待重新验证」的关系，手动触发重新验证
- **删除**：删除不再需要的表关系记录
- **清空**：一键清空当前数据源下所有表关系

#### 3.2.4 缓存失效联动

当用户在「数据查询」模块中刷新表结构（Schema）时：

- 自动将对应数据源的所有表关系标记为 `待重新验证`
- 下次 AI 使用这些关系时会自动重新验证
- 在表关系列表中显示黄色「待验证」标记

### 3.3 数据接入（Excel / 图片 → SQL 语句）

#### 3.3.1 Excel 导入

**支持的文件格式**：`.xlsx`、`.xls`

**操作方式**：用户将 Excel 文件拖拽到「AI报表」页面的输入区域，配合自然语言描述数据处理意图。

**典型场景：**

> 用户拖入 `新药品目录.xlsx`，并输入："把这些药品信息插入到 DRUG_INFO 表"

> 用户拖入 `医保调价.xlsx`，并输入："根据药品编码更新 DRUG_PRICE 表的零售价"

**AI 处理流程：**

1. 解析 Excel 文件内容（Sheet 名、列头、数据行、每列的数据类型）
2. 向用户展示解析结果（前 5 行预览 + 列头信息 + 数据类型推断）
3. 根据用户描述和已缓存的表结构，匹配目标表
4. 生成对应的 SQL 语句：
   - **插入场景**：生成 INSERT INTO ... VALUES (...) 语句
   - **更新场景**：生成 UPDATE ... SET ... WHERE ... 语句
5. 展示生成的 SQL 供用户预览
6. 用户可选择：复制 SQL、导出为 .sql 文件、或在数据库中直接执行

**数据校验要求：**

- 检查 Excel 列与目标表列的匹配情况，不匹配的列高亮提示
- 检查数据类型兼容性（如 Excel 中的文本无法插入数字列）
- 检查必填列是否有空值
- 检查是否存在重复数据（根据主键或唯一约束判断）
- 校验结果在预览时明确展示（通过/警告/错误）

#### 3.3.2 图片导入

**支持的文件格式**：`.png`、`.jpg`、`.jpeg`

**操作方式**：用户将包含表格数据的截图或拍照图片拖拽到输入区域，配合自然语言描述。

**典型场景：**

> 用户拖入一张表格截图，并输入："把这些数据插入到临时统计表 TEMP_STATS"

**AI 处理流程：**

1. 将图片发送给 DeepSeek 多模态模型进行识别
2. 提取图片中的表格结构（列头 + 数据行）
3. 向用户展示识别结果，用户可修正识别错误
4. 后续流程与 Excel 导入一致（匹配目标表 → 生成 SQL → 预览 → 执行/导出）

**识别能力要求：**

- 支持有框线和无框线的表格
- 支持合并单元格的识别
- 识别失败或置信度低时，提示用户手动修正
- 支持用户对识别结果进行编辑（修改单元格值、调整列映射）

#### 3.3.3 执行安全

数据接入生成的 INSERT/UPDATE 语句属于**写操作**，安全要求更高：

- 必须在用户明确确认后才能执行
- 执行前显示影响行数预估
- 支持「模拟执行」模式（只生成 SQL 不实际执行）
- 实际执行后显示执行结果（成功行数、失败行数、错误详情）

### 3.4 报表历史管理

#### 3.4.1 自动保存

每次报表生成完成后，系统自动将以下信息保存为一条历史记录：

- 报表标题（AI 自动生成 + 用户可修改）
- 用户原始描述
- 最终执行的 SQL
- 查询结果数据
- 图表类型和配置
- 完整的对话上下文
- 创建时间

#### 3.4.2 历史记录列表

在「AI报表」页面左侧面板中，按时间分组展示历史记录：

- **今天** / **昨天** / **本周** / **更早**
- 每条记录显示：标题、时间、SQL 摘要（截断前 50 字符）
- 点击记录可回看完整内容（SQL、数据、图表、对话过程）
- 支持搜索：按标题或描述关键词搜索

#### 3.4.3 历史记录操作

| 操作 | 行为 |
|------|------|
| 查看 | 加载完整报表（SQL + 数据 + 图表 + 对话）到右侧展示区 |
| 重跑 | 使用相同的 SQL 重新执行查询（适合查看最新数据） |
| 修改标题 | 双击标题进入编辑模式 |
| 删除 | 删除单条历史记录（需确认） |
| 清空 | 一键清空所有历史记录（需二次确认） |

### 3.5 报表模板

#### 3.5.1 保存为模板

用户可以将一次报表查询保存为模板，便于定期复用：

- 点击「保存为模板」按钮
- 输入模板名称和描述
- SQL 中的**具体值**自动识别为参数（如日期值、科室名等），用占位符替换
- 用户可手动调整哪些部分是参数
- 当前图表类型作为模板的默认图表

#### 3.5.2 模板参数

模板支持的参数类型：

| 参数类型 | 交互控件 | 示例 |
|----------|---------|------|
| 日期 | 日期选择器 | 开始日期、结束日期 |
| 文本 | 输入框 | 科室名称 |
| 下拉选项 | 下拉选择器 | 报表类型（日报/周报/月报） |
| 数字 | 数字输入框 | TOP N 的 N 值 |

#### 3.5.3 模板应用

从模板列表中点击模板：

1. 弹出参数填写面板
2. 用户填写/选择参数值
3. 预览参数替换后的完整 SQL
4. 确认后执行查询并生成报表

#### 3.5.4 模板管理

在左侧面板「报表模板」区域：

- 显示当前项目下所有模板列表
- 支持模板的查看、应用、编辑（修改 SQL/参数/默认图表）、删除
- 模板按创建时间排序，支持搜索

## 五、非功能需求

### 5.1 性能

- SQL 执行超时时间：30 秒（复用现有数据查询模块的超时配置）
- 图表渲染数据量上限：饼图 20 个分类、柱状图 50 个分类、折线图 200 个数据点（超出时给出提示）
- 历史记录列表加载：支持 1000 条记录不卡顿
- Excel 解析：支持 10000 行 × 50 列的 Excel 文件

### 5.2 安全

- SQL 白名单：只允许 SELECT、INSERT、UPDATE（INSERT/UPDATE 仅限数据接入功能，且需用户确认）
- 禁止的 SQL 关键字：DROP、ALTER、TRUNCATE、DELETE、CREATE、GRANT、REVOKE
- DeepSeek API Key：复用全局配置中的加密存储，不额外暴露
- 查询结果中的敏感字段（如密码、身份证号）不做特殊处理（数据本身来自医院内网环境，工具运行在本地）

### 5.3 数据持久化

- 报表历史：存储到本地 SQLite 数据库（复用现有 sql.js 方案）
- 查询结果数据：作为 JSON 字符串存入历史记录
- 表关系缓存：独立存储，Schema 刷新时联动标记失效
- 报表模板：独立存储
- 持久化容量：建议保留最近 90 天的历史记录，超出自动清理

### 5.4 错误处理

- SQL 执行失败：展示错误信息，AI 尝试分析错误原因并建议修正
- 查询结果为空：AI 给出建议（如扩大时间范围、检查筛选条件）
- AI 接口超时/失败：展示友好错误提示，允许重试
- Excel 解析失败：提示文件格式不支持或文件损坏
- 图片识别失败：提示用户尝试更清晰的图片或手动录入

### 5.5 兼容性

- 数据库：Oracle（11g+）、达梦（DM8+）
- 操作系统：Windows 10/11（Electron 桌面应用）
- 图表库：ECharts（CDN 引入或打包内置）

### A. 报表专用 System Prompt 要点

> 以下为 AI 编程时编写 System Prompt 的需求要点，不是完整 Prompt：

1. AI 角色定位：HIS 数据报表助手，帮运维人员将自然语言转为 SQL 和图表
2. SQL 只生成 SELECT，必须限制行数（ROWNUM/TOP 500）
3. 涉及多表必须先验证表关系（查缓存 → 缓存未命中则执行验证 SELECT）
4. 表名和列名必须来自 Schema 缓存，不可编造
5. 图表选择规则：趋势→折线、对比→柱状、占比→饼图、明细→表格、相关性→散点
6. 时间条件使用正确的数据库函数
7. 不确定时主动向用户确认
8. SQL 生成后先展示预览，用户确认后才执行
