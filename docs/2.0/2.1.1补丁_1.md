# 2.1.1 补丁_1 — 数据查询工具修复与优化

## 1. 更新表结构后历史表在页面上消失

**现象**：执行「增量更新」或「覆盖更新」后，页面只显示本次新加载的表结构，之前已存在的表在页面上消失。程序重启后所有表才会正常显示。

**根因分析**：
- `dataSourceStore.ts` 的 `loadSchema` 方法（L200-L265）在加载完成后执行 `set({ schema: processedSchema, ... })`
- 当 `mergeWithExisting=true`（增量更新）时，L259 处的 `set` 直接使用 `processedSchema` 替换了 `state.schema`，而非使用 `state` 回调函数中的合并逻辑
- 具体问题在 L259：`set({ schema: processedSchema, ... })`，这里没有使用 `set((state) => {...})` 的形式，因此合并逻辑（L244-L248）虽然写了但被后续的 `set` 覆盖了
- 覆盖更新时 `mergeWithExisting=false`，直接用新数据替换，这是预期行为，但前端侧栏表列表可能未正确响应 schema 变化

**期望修复**：
- 确保 `mergeWithExisting=true`（增量更新）时，`set` 使用 state 回调函数，正确合并新旧表结构
- 合并逻辑应为：已存在的表更新其列信息，新表追加到列表末尾
- 表名匹配应不区分大小写
- 覆盖更新时，确保前端侧栏表列表能正确响应 schema 的完整替换

**关键代码位置**：
- `loadSchema` 方法：`renderer/stores/dataSourceStore.ts` L200-L265
- 合并逻辑：`renderer/stores/dataSourceStore.ts` L244-L248（当前使用 `set((state) => {...})` 但 L259 处又执行了另一个 `set` 直接覆盖）
- `refreshSchema`（覆盖更新）：`renderer/stores/dataSourceStore.ts` L330-L336
- `refreshSchemaWithMerge`（增量更新）：`renderer/stores/dataSourceStore.ts` L338-L344
- 前端更新按钮：`renderer/pages/Schema.tsx` L1685-L1712

**修复参考**：
将 L259 处的 `set({ schema: processedSchema, ... })` 移除，确保合并逻辑在唯一的 `set` 调用中完成。当前代码存在两次 `set` 调用，第一次（L241）使用了 state 回调做合并，第二次（L259）直接覆盖了结果。应将两次 set 合并为一次，或在第二次 set 中也使用 state 回调保留合并结果。

## 2. 查询条件过多时表格消失且无法滚动

**现象**：当查询条件数量较多时，下方查询结果表格区域被完全挤出可视区域，且无法通过滚动到达表格位置。

**根因分析**：
- 查询面板整体结构为 `flex-1 flex flex-col overflow-hidden`（L1155），查询条件区域使用 `flex-shrink-0`（L1156），表格区域使用 `flex-1 overflow-hidden`（L1512）
- `flex-shrink-0` 导致查询条件区域不会被压缩，当条件过多时该区域无限增长
- 表格区域虽然设置了 `flex-1`，但父容器 `overflow-hidden` 导致超出部分不可见
- 查询条件区域没有设置 `max-height` 和内部滚动，条件多时撑满整个容器
- 查询条件容器设置了 `max-w-[min(100%,42rem)]`（L1203、L1391），限制了宽度但未限制高度

**期望修复**：
- 查询条件区域（L1203 的 div）添加 `max-h-[16rem]` 或类似的最大高度限制，并设置 `overflow-y-auto` 使条件过多时可内部滚动
- 排序条件区域同理，如条件过多也应支持内部滚动
- 确保表格区域始终有最小可见高度（如 `min-h-[12rem]`），不会被完全挤出
- 整体查询面板的布局应保证：条件区域可滚动 + 表格区域始终可见

**关键代码位置**：
- 查询面板容器：`renderer/pages/Schema.tsx` L1154-L1155 `flex-1 flex flex-col overflow-hidden`
- 查询条件区域（flex-shrink-0）：`renderer/pages/Schema.tsx` L1156
- 查询条件内容容器（max-w 限制）：`renderer/pages/Schema.tsx` L1202-L1203
- 排序条件容器：`renderer/pages/Schema.tsx` L1391
- 表格区域：`renderer/pages/Schema.tsx` L1512 `flex-1 overflow-hidden`

## 3. 排序条件取消光标自动定位到添加字段输入框

**现象**：添加排序条件后，光标会自动定位到「+ 添加字段」输入框，用户不期望此行为。

**根因分析**：
- `addSortCondition` 方法（L483-L488）在添加条件后调用了 `setSortDropdownOpen(prev => ({ ...prev, [tableName]: false }))` 关闭下拉
- 但添加排序条件后，输入框仍保持 focus 状态（因为用户是从下拉列表点击或回车确认的），导致光标停留在输入框中
- 查询条件的添加方法（`addQueryCondition`）也有类似行为，但查询条件的输入框 focus 是用户主动触发的

**期望修复**：
- 在 `addSortCondition` 方法中，添加条件后主动将焦点从输入框移除
- 可通过 `document.activeElement` 获取当前焦点元素并调用 `blur()`，或通过 ref 操作输入框
- 具体实现：在 `addSortCondition` 末尾添加 `(document.activeElement as HTMLElement)?.blur()`

**关键代码位置**：
- `addSortCondition` 方法：`renderer/pages/Schema.tsx` L483-L488
- 排序输入框：`renderer/pages/Schema.tsx` L1452-L1480

## 4. 查询条件右侧空间利用 — 改为左右两栏布局

**现象**：查询条件区域当前限制最大宽度为 `42rem`（L1203、L1391），右侧大片空白未被利用。查询条件和排序条件上下排列，整体区域偏窄。

**期望布局调整**：
- 将查询条件和排序条件改为左右两栏布局：
  - **左栏**：查询条件（占约 60-65% 宽度）
  - **右栏**：排序条件（占约 35-40% 宽度）
- 移除查询条件和排序条件的 `max-w-[min(100%,42rem)]` 宽度限制
- 两栏之间使用分隔线或间距区分
- 响应式处理：在窄屏（如 < 768px）时回退为上下排列

**布局参考**：
```
┌─────────────────────────────────────────────────────────┐
│  [✓ 列名显示中文]  [已保存条件: xxx]                      │
├──────────────────────────┬──────────────────────────────┤
│  查询条件                 │  排序条件                      │
│  ┌──────────────────┐    │  ┌────────────────────────┐   │
│  │ 条件1: 字段 = 值  │    │  │ 字段A ↑  字段B ↓  +添加 │   │
│  │ 条件2: 字段 > 值  │    │  └────────────────────────┘   │
│  │ + 添加字段        │    │                              │
│  └──────────────────┘    │                              │
├──────────────────────────┴──────────────────────────────┤
│  [执行查询]  [导出SQL]  [导出Excel]                       │
├─────────────────────────────────────────────────────────┤
│  查询结果表格                                             │
└─────────────────────────────────────────────────────────┘
```

**关键代码位置**：
- 查询条件容器（当前 max-w 42rem）：`renderer/pages/Schema.tsx` L1202-L1203
- 排序条件容器（当前 max-w 42rem）：`renderer/pages/Schema.tsx` L1391
- 查询面板整体布局：`renderer/pages/Schema.tsx` L1154-L1510
- 操作按钮区域（fixed 定位）：`renderer/pages/Schema.tsx` L1190-L1200（需改为非 fixed 定位，放入两栏布局下方或上方工具栏中）

**注意事项**：
- 操作按钮（执行查询、导出SQL、导出Excel）当前使用 `fixed top-32 right-6` 定位（L1190），改为两栏布局后应调整为流式布局，放在条件区域上方或下方
- 查询条件和排序条件的「+ 添加字段」输入框及下拉列表需确保在两栏布局中不被截断
- 两栏各自应有独立的 `max-height` 和 `overflow-y-auto`，防止条件过多时撑破布局
