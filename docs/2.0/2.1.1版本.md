# 数据查询工具优化需求

## 项目背景
这是一个基于Electron的数据查询与AI报表工具，支持Oracle和达梦数据库。主要功能包括表结构管理、数据查询和AI分析。

## 需要优化的问题

### 1. 表结构增量更新问题
**现象**：更新表结构时，增量更新变成了覆盖更新，导致之前的表结构消失。

**期望**：
- 增量更新应该保留现有表结构，只添加新表
- 当表已存在时，应该更新表结构信息而不是覆盖
- 确保表名匹配逻辑正确（不区分大小写）

### 2. 表结构加载进度显示
**现状**：已有进度条UI，但需要优化。

**期望**：
- 细化加载进度阶段（获取表列表、加载表结构等）
- 显示更详细的进度信息（当前操作、表名等）
- 优化进度条动画，提升用户体验

### 3. 查询条件UI布局问题
**现象**：数据查询的查询条件相关操作，会跟随数据表的列宽跑到最右边。

**期望**：
- 查询条件区域应有最大宽度限制
- 确保查询条件始终在可视区域内
- 优化响应式设计，适应不同屏幕尺寸

### 4. 表结构更新时自动过滤无注释表
**现象**：数据库中存在大量表注释（comments）为空的表（如临时表、中间表），加载表结构时这些表也会被加载，导致表列表冗长、干扰用户查找有效表。

**期望**：
- 增量更新和覆盖更新两种模式下，都自动过滤表注释（comments）为空的表
- 过滤逻辑应在后端数据库查询层面实现（`oracle.ts` 和 `dameng.ts`），而非前端过滤
- 当前已有 `filterEmptyTables` 参数用于过滤无数据表，新增过滤无注释表应与之并列，可考虑统一为 `filterRules` 或新增独立参数如 `filterNoCommentTables`
- Oracle 查询中表注释来源为 `all_tab_comments`，达梦类似；需注意 `NVL(comments, ' ')` 将空注释转为空格的情况，判断时应 trim 后检查
- 过滤行为应在 IPC 层（`handlers.ts`）透传参数，前端 `dataSourceStore.ts` 的 `loadSchema` 方法需支持传入该选项
- 在 Schema 页面的加载表结构 UI 中，提供对应的勾选开关，让用户可选择是否过滤无注释表（默认勾选）

**关键代码位置**：
- Oracle 表注释查询：`main/database/oracle.ts` L152 `SELECT owner, table_name, NVL(comments, ' ') as table_comments FROM all_tab_comments`
- 达梦表注释查询：`main/database/dameng.ts` L219-L222
- 表结构过滤入口：`main/database/oracle.ts` L285 `if (!filterEmptyTables || hasTableData)`
- IPC 参数传递：`main/ipc/handlers.ts` L185 `db:getSchema`
- 前端加载方法：`renderer/stores/dataSourceStore.ts` L190 `loadSchema`

### 5. 数据查询排序组件优化
**现状**：当前排序组件采用输入框+下拉列表添加字段、`<select>` 切换 ASC/DESC、X 按钮删除的方式，交互不够直观，UI 较为简陋。

**当前实现**：
- 排序条件数据结构：`{ columnName: string; order: 'ASC' | 'DESC' }[]`
- 添加方式：输入框输入字段名或从下拉列表选择，回车确认添加
- 排序方向切换：`<select>` 下拉选择 ASC/DESC
- 删除方式：X 按钮逐个删除
- UI 样式：简单的 flex 横向排列，白色背景圆角卡片

**期望优化方向**：
- **拖拽排序**：支持拖拽调整排序条件的优先级顺序（即 SQL ORDER BY 中字段的先后顺序）
- **点击切换排序方向**：点击排序方向按钮即可在 ASC ↔ DESC 之间切换，替代当前的 `<select>` 下拉，更快捷直观
- **UI 美化**：
  - 排序条件标签采用更紧凑的 pill/tag 样式
  - ASC 用 ↑ 图标、DESC 用 ↓ 图标，视觉更直观
  - 拖拽手柄使用 ⋮⋮ 或 ≡ 图标提示可拖拽
  - 删除按钮保持 X 图标，hover 时高亮


**关键代码位置**：
- 排序条件类型定义：`renderer/pages/Schema.tsx` L23-L26 `SortCondition`
- 排序条件状态管理：`renderer/pages/Schema.tsx` L111 `sortConditions` useState
- 添加/删除/更新方法：`renderer/pages/Schema.tsx` L480-L506
- SQL 构建逻辑：`renderer/pages/Schema.tsx` L528-L531
- 排序 UI 渲染：`renderer/pages/Schema.tsx` L1359-L1434

### 6. 查询条件中文名列显示问题
**现象**：当未勾选「列名显示中文」时，查询条件中每个条件项仍然会显示两行——第一行是英文字段名，第二行本应显示中文注释的位置也显示了英文字段名，造成信息重复。

**当前逻辑分析**：
- `getColumnDisplayName(column)` 方法（L616-L621）：当 `showColumnNamesInChinese=true` 且 `column.comments` 存在时返回 comments，否则返回 columnName
- 查询条件中列名显示（L1296-L1305）：
  - 第一行始终显示 `condition.columnName`（英文字段名）
  - 第二行通过 `column?.comments` 判断是否有注释，有则显示 `getColumnDisplayName(column)`
  - 问题在于：当 `showColumnNamesInChinese=false` 时，`getColumnDisplayName` 始终返回 `columnName`，导致第二行也显示英文字段名
- 结果表格表头（L1465）：已正确处理，未勾选时直接显示英文字段名

**期望**：
- 当未勾选「列名显示中文」时，查询条件中只显示英文字段名一行，隐藏中文注释行
- 当勾选「列名显示中文」时，查询条件中显示英文字段名+中文注释两行（保持现有行为）
- 修改位置：`renderer/pages/Schema.tsx` L1300-L1304，将 `column?.comments` 的判断条件改为同时检查 `showColumnNamesInChinese` 状态

**修复参考**：
```
// 当前代码（L1300-L1304）
{column?.comments ? (
  <div className="truncate text-[10px] text-slate-400" title={column.comments}>
    {getColumnDisplayName(column)}
  </div>
) : null}

// 期望修改为
{showColumnNamesInChinese && column?.comments ? (
  <div className="truncate text-[10px] text-slate-400" title={column.comments}>
    {column.comments}
  </div>
) : null}
```

### 7. 表结构存储解耦与稳定性
**现象**：当前表结构数据（schema_cache）存储在 SQLite 数据库文件（`userData/zoehis-helper.db`）中，与数据源配置、查询历史等其他业务数据共用同一个数据库文件。当代码进行大规模重构或架构改造时，SQLite 表结构变更（如 `DROP TABLE`、`CREATE TABLE`）容易导致 schema_cache 数据丢失，而表结构数据是从远程数据库耗时加载的，丢失后重新加载成本很高。

**当前存储机制分析**：
- 存储位置：`app.getPath('userData')/zoehis-helper.db`（sql.js 内存数据库 + 文件持久化）
- schema_cache 表结构：`id, dataSourceId, schemaData(JSON), filterPattern, cachedAt`
- 已有迁移逻辑（`sqlite.ts` L101-L166）：在 `data_sources` 表增加 `projectId` 列时，会先保存 schema_cache 数据再恢复，但此迁移逻辑脆弱，仅覆盖了特定场景
- schemaData 字段存储完整的 `TableInfo[]` JSON，数据量可能很大
- 前端 Zustand store 的 `partialize`（L447-L457）中未持久化 schema 数据，schema 仅在内存中，依赖后端缓存

**期望**：
- **将表结构缓存独立存储**：将 schema_cache 从主 SQLite 数据库中分离，使用独立的 JSON 文件存储，路径如 `userData/schema-cache/{dataSourceId}.json`
- **文件结构设计**：
  - 每个数据源一个独立的 JSON 文件，文件名为 `{dataSourceId}.json`
  - JSON 内容包含：`{ version: string, dataSourceId: string, cachedAt: string, tables: TableInfo[] }`
  - `version` 字段用于未来格式变更时的兼容性判断
- **读写方式**：
  - 写入：表结构加载完成后，通过 `fs.writeFileSync` 将 JSON 写入对应文件
  - 读取：切换数据源时，通过 `fs.readFileSync` 读取对应文件，文件不存在则返回空
  - 删除：删除数据源时，同步删除对应的缓存文件
- **优势**：
  - 与主数据库完全解耦，代码重构修改 SQLite 表结构时不会影响表结构缓存
  - 每个数据源独立文件，读写互不影响，性能更好
  - JSON 文件可读性好，便于调试和手动备份
  - 天然支持版本号字段，未来格式升级时可按版本迁移
- **迁移策略**：
  - 首次启动时检测 `userData/schema-cache/` 目录是否存在，不存在则创建
  - 检测旧的 SQLite schema_cache 中是否有数据，有则迁移到 JSON 文件后清除旧表数据
  - 迁移完成后可保留 schema_cache 表定义（空表），避免 SQLite 报错

**关键代码位置**：
- SQLite 初始化与迁移：`main/database/sqlite.ts` L86-L170
- schema_cache 表定义：`main/database/sqlite.ts` L211-L220
- 缓存读取：`main/database/sqlite.ts` L519 `getSchemaCache`
- 缓存写入：`main/database/sqlite.ts` L599 `setSchemaCache`
- 缓存删除（单表/多表）：`main/database/sqlite.ts` L644-L675
- IPC 缓存接口：`main/ipc/handlers.ts` L175-L236
