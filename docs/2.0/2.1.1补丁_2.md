# 2.1.1 补丁_2 — 数据查询布局优化与项目更名

## 1. 数据查询界面布局与滚动优化

### 现象
- 数据查询页面中，查询结果表格的展示区域过小，大约只能显示两行数据
- 查询条件区域和排序条件区域各自有独立的内部滚动，体验割裂
- 整个查询页面没有全局滚动，条件多时表格被压缩到极小

### 期望
- **整个数据查询页面支持全局滚动**：将查询条件、排序条件、查询结果表格放在同一个可滚动容器中，用户可以自然地上下滚动浏览所有内容
- **移除查询条件和排序条件的内部滚动**：条件区域不再限制高度和独立滚动，条件多时自然撑开高度，由页面全局滚动来浏览
- **查询结果表格区域不再被压缩**：表格应有足够的展示高度（如 `min-h-[20rem]`），不再因为条件区域占满空间而被挤压到只剩两行
- **一键置顶按钮**：页面滚动后，右下角显示「回到顶部」按钮，点击后平滑滚动到页面顶部，方便快速回到查询条件区域
- **优化滚动条样式**：自定义滚动条外观，使用更细、更现代的滚动条样式（如 6px 宽度、圆角、hover 时显示、默认半透明），替代浏览器默认滚动条

### 布局思路
当前布局采用 CSS Grid 三行结构（操作栏 / 条件+排序 / 结果表格），通过 `overflow-hidden` 将空间约束在视口内，导致表格被压缩。期望改为：页面整体可滚动，条件区域自然撑高，表格区域有足够最小高度，用户通过全局滚动在条件和结果之间切换。

### 关键代码位置
- 查询页面 Grid 容器：`renderer/pages/Schema.tsx` L1156（当前 `grid-rows-[auto_minmax(0,1fr)_minmax(12rem,1fr)] overflow-hidden`）
- 条件+排序行容器：`renderer/pages/Schema.tsx` L1201（当前 `overflow-hidden`）
- 查询条件内容区：`renderer/pages/Schema.tsx` L1285（当前 `overflow-y-auto max-h-[min(16rem,42vh)]`）
- 排序条件内容区：`renderer/pages/Schema.tsx` L1399（当前 `overflow-y-auto max-h-[min(16rem,42vh)]`）
- 查询结果区域：`renderer/pages/Schema.tsx` L1517（当前 `overflow-hidden`）
- 查询结果表格滚动区：`renderer/pages/Schema.tsx` L1532（当前 `overflow-auto`）

## 2. 项目更名 ZoeDevOps

### 现象
当前项目名称为 `zoehis-helper`，需要统一更名为 `ZoeDevOps`。

### 期望
将所有涉及项目名称的位置统一更改为 `ZoeDevOps`，包括但不限于：

| 类别 | 当前值 | 期望值 |
|------|--------|--------|
| 包名（package.json name） | `zoehis-helper` | `zoe-devops` |
| 产品名（package.json productName） | `zoehis-helper` | `ZoeDevOps` |
| 应用ID（package.json appId） | `com.zoehis.helper` | `com.zoe.devops` |
| 窗口标题（main/index.ts） | `zoehis-helper` | `ZoeDevOps` |
| 关于对话框（main/index.ts） | `zoehis-helper v1.0` | `ZoeDevOps v1.0` |
| HTML 页面标题（renderer/index.html） | `zoehis-helper` | `ZoeDevOps` |
| 数据库文件名（main/database/sqlite.ts） | `zoehis-helper.db` | `zoe-devops.db` |
| 加密密钥标识（main/database/sqlite.ts） | `zoehis-helper-encryption-key-v1` | `zoe-devops-encryption-key-v1` |
| Zustand 存储 key（各 store 文件） | `zoehis-helper-storage` 等 | `zoe-devops-storage` 等 |

### 注意事项
- 数据库文件名变更后，需考虑旧数据库文件的迁移（首次启动时检测旧文件 `zoehis-helper.db` 是否存在，存在则重命名为 `zoe-devops.db`）
- Zustand 存储 key 变更后，localStorage 中的旧数据会丢失，需考虑迁移或接受用户重新配置
- 加密密钥变更后，旧密钥加密的密码将无法解密，需在迁移时用旧密钥解密后再用新密钥加密

- package.json：`package.json` L2（name）、L51（appId）、L52（productName）
- Electron 窗口标题：`main/index.ts` L83
- 关于对话框：`main/index.ts` L63
- HTML 标题：`renderer/index.html` L6

## 3. AI 配置更名为全局配置，整合 GitLab Token

### 现象
- 当前「AI 配置」Tab 仅包含 DeepSeek 的 API Key、Base URL、Model 三项配置
- GitLab Token 的配置链路是断裂的：
  - 全局 `GITLAB_CONFIG` 对象中 token 初始为空，且仅存在于内存中，应用重启后丢失
  - IPC 通道 `ai:setGitLabConfig` 虽已定义但是空实现，前端调用后不会实际更新 token
  - 仓库级别的 `gitLabToken` 字段在数据库中已实现加密存储，但前端表单中缺少输入项，用户无法配置
  - 实际使用时 token 取值逻辑为 `repoConfig.gitLabToken || GITLAB_CONFIG.token`，两者都可能为空

### 期望
- **「AI 配置」更名为「全局配置」**：Tab 标签文字从"AI 配置"改为"全局配置"，定位为应用级别的统一配置入口
- **新增 GitLab Token 配置项**：在全局配置表单中增加 GitLab 相关配置：
  - GitLab Base URL（默认值 `http://gitlab.zoesoft.com.cn`）
  - GitLab Token（密码输入框，加密存储）
- **替换硬编码**：将 `main/agent/config.ts` 中 `GITLAB_CONFIG` 的硬编码 baseUrl 和 token 替换为从全局配置（`global_config` 表）中读取
- **修复 IPC 空实现**：补全 `ai:setGitLabConfig` IPC handler 的实际逻辑，使其调用 `updateGitLabConfig()` 更新内存中的全局配置
- **Token 优先级保持不变**：使用时仍遵循「仓库级 token 优先，全局 token 回退」的逻辑

### 全局配置表单期望结构
```
全局配置
├── DeepSeek
│   ├── API Key（密码输入框）
│   ├── Base URL（默认 https://api.deepseek.com/v1）
│   └── Model（默认 deepseek-chat）
└── GitLab
    ├── Base URL（默认 http://gitlab.zoesoft.com.cn）
    └── Token（密码输入框）
```
